## 3. PARTIAL / FAIL 根因分析

### 3.1 `invoiceNumber: null` / `allowanceNumber: null`（S1.1、S1.3、S3.2）

| 層 | 行為 |
| --- | --- |
| n8n workflow | 把 body 送進 svc — OK |
| svc (Hono) | 打 sandbox `/json/f0401` — OK |
| sandbox `/json/f0401` | 回 200 但 body 不含 MIG response 該有的 `InvoiceNumber` 欄位 |
| svc 解析 | `result.InvoiceNumber` undefined → 回上層 `invoiceNumber: null` |

**這是 sandbox 模擬器待補功能**，不是 workflow 或 svc 缺陷。生產環境（真實 Amego）此值會正常回填。

### 3.2 Bad request 該被拒卻回 200（S1.2、S3.2、S5.2）

sandbox 模擬器目前只接收 + log，不做欄位驗證。生產環境（真實 Amego / svc）會驗 `salesAmount + taxAmount == totalAmount`、`invoiceNumber` 格式等。

> **note**：v0.30.2 加進 svc 的 `op` 白名單驗證（SEC-2）仍生效於真實流量，但 S5.2 因 sandbox 寬容沒走到那層。

### 3.3 S2.1 void seed 抓不到 invoiceNumber

void workflow 含 Wait node — webhook 不會立刻回應。本次把 seed POST 改成 fire-and-forget（搭配 `AbortSignal.timeout(2500)`）避開 fetch headers-timeout crash，但代價是測試腳本拿不到 seed 的 `invoiceNumber`。**workflow 本身正確**（execution 停在 Wait，符合 human-approval gate 設計）。

未來修法：sim-runner 改用 n8n executions API 取得 seed execution 的 stamp output，而非等 webhook response。

### 3.4 twin-node test injection 表現

S4.1 與 S6.1 是兩個**完全沒有 Google Sheets credential** 的環境跑出來的：

| scenario | 預期 | 實際 |
| --- | --- | --- |
| S4.1 reconcile | 讀 3 筆假 audit row → filter 出昨天 → svc /v1/query → 比對 → mismatch 1 | `total=3 green=2 mismatch=1` ✓ |
| S6.1 monthly export | 讀 5 筆假上月 row → render CSV → Convert to File → 不寄信 | `rowCount=5 counts={issue:3,void:1,allowance:1}` ✓ |

**結論**：`[TEST] Read Audit` Code 節點當 Google Sheets 節點的 sibling，toggle `disabled` flag 即切換假資料/真資料，**SKILL §1.7 模式可以 commit**。
