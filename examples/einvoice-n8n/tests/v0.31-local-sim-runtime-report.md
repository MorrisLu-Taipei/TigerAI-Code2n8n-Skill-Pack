## 2. Why not use the local sandbox for the SDK happy-path tests

The local sandbox routers reach the right paths now (Amego f0401/f0501/g0401/g0501, ECPay /B2CInvoice/Issue etc.) but the payload-parsing in each router was written from API-shape guesses, not from a full reading of each vendor's MIG 4.0 form specification. The Amego f0401 router, for example, accepts the request but currently looks for `OrderId`/`orderId` in the body, whereas the SDK sends a MIG-encoded form with field names like `Invoice`, `BuyerIdentifier`, etc. Bringing each router to full MIG fidelity is on the v0.32 roadmap; for v0.31.0, the pragmatic split is:

| What | Where it ran |
| --- | --- |
| SDK happy-path (real Amego invoice issued / voided / queried) | Real Amego public sandbox (Amego's hosted environment) |
| SDK happy-path (other 4 vendors) | Not run end-to-end — capability check only (see §3.B) |
| Failure injection (5xx / auth-fail / quota-exhausted) | Local Amego sandbox endpoint via `X-Sandbox-Inject` header — the route is reached and the header is honored, demonstrating the injection mechanism is functional |
| Google Sheet append / read | Local Sheet sim — verified directly |
| Slack chat.postMessage | Local Slack sim — verified directly |
| svc trust boundary (401 / 413 / 400 / config-error) | Local svc — verified directly |

### 3.E — Sandbox failure-injection mechanism

These call the sandbox Amego endpoint directly (bypassing svc + SDK) with various `X-Sandbox-Inject` headers, to demonstrate the injection mechanism that the v0.32+ work will use once the MIG payload mapping is complete.

| Inject code | Expected sandbox response | Observed | Status |
| --- | --- | --- | --- |
| `5xx` | HTTP 502 with vendor-shaped body `{"code":"500","msg":"internal"}` | HTTP 502 + matching body | PASS |
| `auth-fail` | HTTP 200 + Amego `{"code":"99","msg":"Auth failed"}` (Amego returns 200 with error code in body) | HTTP 200 + matching body | PASS |
| `quota-exhausted` | HTTP 200 + `{"code":"16","msg":"字軌耗盡"}` | HTTP 200 + matching body | PASS |
| `not-found` | HTTP 200 + `{"code":"4","msg":"找不到發票"}` | HTTP 200 + happy-path response (sandbox couldn't determine which invoice to "not find" because MIG payload mapping is incomplete) | PARTIAL — header path reached, response wrong because of the v0.32 MIG-mapping gap |
| `conflict` | HTTP 200 + `{"code":"13","msg":"已作廢"}` | HTTP 200 + happy-path response | PARTIAL — same root cause as not-found |

Confirmed: the header is read, the inject codes are recognized for stateless injections (5xx / auth / quota), and the mechanism is wired correctly. For stateful injections (not-found / conflict, which depend on the sandbox knowing which invoice the request refers to) the gap is in the MIG payload parser, not in the inject mechanism.

### 3.H — Workflows not exercised end-to-end via n8n

Five of the six einvoice workflows have NOT been executed inside n8n. They were verified at the structural layer only (scanner clean, REST import 6/6 ok). Reasons:

- `einvoice-void-with-approval`: requires Slack OAuth credential, or the Wait-resume webhook to be wired up to an externally accessible URL.
- `einvoice-allowance`: same shape as issue; depends on a previously-issued invoice and a real allowance flow.
- `einvoice-daily-reconcile`: scheduled trigger; requires either a manual canvas "Execute workflow" click or several days of audit-row history.
- `einvoice-provider-failover`: a sub-workflow, called by other workflows; requires `executeWorkflow` wiring and at least two configured providers.
- `einvoice-monthly-audit-export`: scheduled trigger on the 1st of month; requires audit history.

These are tracked for v0.32+ as "structurally clean, runtime PENDING".

## 4. n8n behavior re-confirmed (`code2n8n-pipeline` SKILL Stage 10 caveat)

The "REST API workflow creation does not auto-register webhook listeners" finding from v0.30.2 was reconfirmed in v0.31.0 testing: the workflow imported via REST `POST /workflows` + activated via `POST /workflows/{id}/activate` shows `active: true` in the DB but `POST /webhook/<path>` returns 404 with the hint "The workflow must be active for a production URL to run successfully." A single Save click in the n8n UI is what registers the webhook with the listener. This continues to be documented in `code2n8n-pipeline` SKILL Stage 10.
