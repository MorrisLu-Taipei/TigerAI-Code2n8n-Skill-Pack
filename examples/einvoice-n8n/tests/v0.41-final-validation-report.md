## 6. SEC-022（meta-lesson）— 為何 docker vendor 模擬器是 over-engineering

### 6.1 背景

v0.30.1 系列為了「離線跑 workflow 端到端」自蓋了 docker vendor 模擬器：5 個 router 假裝是 Amego / ECPay / ezPay / ezPay-CB / ezReceipt 的 HTTP API。

### 6.2 為何不可靠

自蓋 stub 本質有 4 個結構問題：

| 面向 | 自蓋 docker stub |
| --- | --- |
| Zod schema 驗證 | 我們手寫，跟真實 SDK adapter **會漂移**（沒有同步機制） |
| 狀態機（void 後拒 void） | 沒實作 |
| capabilities 強制（SCHEDULED_ISSUE on Amego 拒絕）| 沒實作 |
| 不需網路 | 要起 docker container |

**對 Amego**：用真實 sandbox 才是 ground truth。**對其他 4 家**：無公開測試帳號 → **runtime 未驗、誠實揭露**，不靠自蓋 stub 替代信心（自蓋 stub 跑通 ≠ 真實環境跑通）。

### 6.3 為何當初寫了

v0.30.1 時 implementing AI（我）寫 SKILL §8 sandbox build directive 時**沒讀完 SDK README** — 看到「testing without credentials」就自己刻 docker stub。**同 v0.40.0 的「3 個偽 PASS」教訓**：沒先讀完 source 就動手，繞遠路。

### 6.4 v0.41.0 處置

- SKILL §8 sandbox build directive 改為**先查 SDK README 是否提供 mock 機制**，有則用、無則才刻 stub
- `examples/einvoice-n8n/sandbox/src/routers/{amego,ecpay,ezpay,ezpay-cb,ezreceipt}.ts` 標 `@deprecated` 註解
- 本地 Email / Sheet / Slack 模擬器**保留**（這些 SDK 沒有，且解決外部服務問題）
- 對外宣稱：**本 case study 不需要 docker vendor 模擬器**；其他 4 家 runtime 未驗 — 誠實揭露

### 6.5 元教訓寫進 SKILL 持久層

下一個 case study 啟動 Stage 8（sandbox build）時，main agent **必須**先讀 SDK README + 看 testing section 是否提供 mock 機制。critic gate（lexical regex）偵測「`stub` / `simulator` / `mock`」字眼於 SKILL §8 範圍 → 要求 evidence 證明已查過 SDK。同 §1.6 / §1.8 同級不可繞。
